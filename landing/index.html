1. Visión general

TPDroid es una aplicación de escritorio web que se conecta a dispositivos Android vía USB (ADB embebido) para gestionar procesos en ejecución y apps instaladas. Permite forzar detención de procesos, deshabilitar/activar notificaciones, y aplicar bloqueo profundo en juegos.

El frontend está embebido en el binario Go via go:embed. Un solo binario, cero dependencias externas. Backend con Go/Gin, frontend con Tailwind CSS (CDN) y JS vanilla.

Incluye un sistema de licencias de 4 capas usando Cloudflare Workers y Supabase, y un panel de administración FastAPI para gestionar códigos de licencia.

Arquitectura del proyecto

backend/           — Servidor Go/Gin con frontend embebido
  adb/              — Wrappers de comandos ADB
  handlers/         — Handlers HTTP (status, device, processes, apps, ads)
  frontend/         — UI embebida (HTML, CSS, JS)
  licensing/        — Paquete de validación de licencias
cloudflare/         — Worker + schema SQL de Supabase
activator/          — Binario de activación de licencias
api_admin/          — Panel de administración (FastAPI)

2. Instalación

Prerrequisitos

  • Go 1.21+ (solo para compilación)
  • Dispositivo Android con Depuración USB habilitada
  • Cable USB
  • No requiere ADB instalado por separado

Modo desarrollo

git clone https://github.com/pabloacisera/TPdroid---adb-manager.git && cd TPdroid---adb-manager
./script/dev.sh
# Se abre http://localhost:8080

O manualmente:

cd backend && go run main.go

Build por plataforma

make build-linux     # → dist/android-manager-linux
make build-windows   # → dist/android-manager-windows.exe
make build-macos     # → dist/android-manager-macos
make build-all       # → todos

El binario incluye el frontend embebido. Se copia a cualquier máquina, se ejecuta, se abre http://localhost:8080.

3. ADB y conexión

Preparar el teléfono

  1. Ajustes → Acerca del teléfono → tocar Número de compilación 7 veces
  2. Ajustes → Opciones de desarrollador → activar Depuración USB
  3. Mantener el teléfono desbloqueado

Conectar

  1. Conectar el teléfono por USB
  2. La app detecta el dispositivo automáticamente (polling cada 2s, timeout 60s)
  3. Revisar la pantalla del teléfono y pulsar "Permitir depuración USB"
  4. Marcar "Recordar siempre" para no repetirlo

Heartbeat

La app mantiene un heartbeat cada 2s con detección de desconexión y banner con countdown. Usa Page Visibility API para pausar polling cuando la pestaña no está visible.

4. Dashboard — Procesos

Lista de procesos en ejecución con PID, nombre, UID y estado (R/S/I/Z/T/D).

  • Usar "Running only" para filtrar solo procesos activos
  • Búsqueda en tiempo real con debounce
  • Paginación client-side (25 procesos por página)
  • Botón Force Stop para detener procesos de usuario

Protección de procesos del sistema

Un proceso se considera del sistema si su UID es system, root o shell, o su package empieza con com.android., android. o es android. Los botones aparecen deshabilitados en la UI y el backend retorna HTTP 403.

Excepción: Chrome (com.android.chrome) y otros browsers instalables son bloqueables aunque usen el prefijo com.android.*. La función IsSystemProcess() tiene una allowlist explícita que los excluye.

5. Dashboard — Apps y Notificaciones

Lista de apps de usuario con paquete, etiqueta y permisos.

  • Búsqueda por nombre/paquete en tiempo real
  • Botón para Desactivar/Activar notificaciones
  • En juegos: se muestra badge y se aplica Triple-Lock profundo
  • Permisos truncados (max 2) con botón "ver" para listar todos en modal
  • Paginación client-side (20 apps por página)

6. Detección de juegos

TPDroid detecta juegos automáticamente usando tres estrategias en orden:

1. Package name analysis

Busca segmentos como .game, .games, .gaming, game., games. y sufijos game/games.

2. Game engines & Ad SDKs

Detecta packages que contengan: unity, unreal, cocos, ironsource, applovin, vungle, admob, chartboost, tapjoy.

3. Permission combo

Si una app tiene VIBRATE + INTERNET, se considera juego (alta probabilidad).

Exclusion list

Contiene ~400 prefijos de packages conocidos no-juego (WhatsApp, Facebook, Chrome, Gmail, etc.). Si un package coincide, se descarta como juego inmediatamente.

Categorías cubiertas: Social & Communication, Browsers, Google Apps, Microsoft, Adobe, Streaming, Banking & Finance, Shopping, Food Delivery, Productivity, Health & Fitness, Maps & Travel, Utilities, VPN & Remote, Amazon Apps, Samsung, Xiaomi (MIUI), Huawei, OnePlus, Oppo, Vivo, Realme, y otros OEMs.

7. Triple-Lock para juegos (Android 13+)

Cuando se deshabilitan notificaciones de un juego, se aplican estos 4 comandos:

appops set POST_NOTIFICATION deny
pm revoke POST_NOTIFICATIONS
pm set-permission-flags user-set
pm clear-permission-flags user-fixed

Al re-habilitar:

appops set POST_NOTIFICATION allow
pm grant POST_NOTIFICATIONS
pm set-permission-flags user-set
pm clear-permission-flags user-fixed

Los errores de pm se silencian (Android 12- no soporta estos flags). La UI muestra un toast confirmando el bloqueo profundo.

8. Ads Scanner

Detecta actividad publicitaria en tiempo real en el dispositivo. El bloqueo de notificaciones por package no es suficiente porque una app legítima puede contener SDKs de publicidad que generan notificaciones u overlays.

Fuentes de detección

SeñalComando ADBQué detecta
active_notifdumpsys notification --noredactNotificaciones activas ahora mismo
web_pushdumpsys notification --noredactNotificaciones push de navegador
ad_alarmdumpsys alarmAlarmas repetitivas con tags de SDKs
overlay_windowdumpsys window windowsVentanas TYPE_APPLICATION_OVERLAY activas

Comando de bloqueo

appops set <pkg> SYSTEM_ALERT_WINDOW deny
appops set <pkg> POST_NOTIFICATION deny
pm revoke <pkg> android.permission.POST_NOTIFICATIONS
am force-stop <pkg>

Bloqueo inteligente por canal (SDK-aware)

El sistema detecta la versión del SDK del dispositivo y adapta la estrategia:

SDK (Android)Estrategia
≥ 26 (Android 8+)cmd notification disable-channel — bloquea solo el canal intrusivo
< 26 (Android 7-)appops set POST_NOTIFICATION deny — bloqueo total

Limitaciones conocidas

  • dumpsys notification --noredact requiere ADB debug
  • En Android 12+ algunas notificaciones tienen texto redactado
  • Alarmas sin tags explícitos pueden no detectarse
  • El bloqueo de Chrome bloquea TODAS las notificaciones web, no solo las publicitarias
  • Tras force-stop, si el usuario abre la app, el SDK puede re-registrar alarmas

Endpoints

MétodoRutaDescripción
GET/api/ads/scanEjecuta los 3 dumpsys, parsea y devuelve lista unificada
POST/api/ads/blockRevoca overlay + bloquea notificaciones + force-stop (acepta canales y SDK)
POST/api/ads/unblockRestaura overlay + reactiva notificaciones (simétrico)
POST/api/ads/block-fullBloqueo total explícito, ignora canales y SDK

9. Sistema de licencias

DRM de 4 fases para el binario compilado TPDroid, usando Cloudflare Workers y Supabase.

Arquitectura

┌──────────────┐   POST /activar   ┌──────────────┐   REST   ┌──────────┐
│  activator   │ ────────────────→ │   Worker     │ ────────→│ Supabase │
│  (Win/Linux) │ ←──────────────── │  cloudflare/  │ ←─────── │          │
└──────────────┘   .lic (firmado)   └──────┬───────┘          └──────────┘
┌──────────────┐   POST /revalidar        │
│   backend    │ ──────────────────────────→
│  (Go/Gin)    │ ←──────────────────────────
└──────┬───────┘   ok / error
       │ 403 si no hay licencia
       ▼
┌──────────────┐
│ license-     │
│ error.html   │
└──────────────┘

Flujo de activación

  1. El desarrollador genera un código en el Panel Admin → se guarda en Supabase
  2. El cliente ejecuta TPDroid-Setup.exe (o install.sh en Linux)
  3. El instalador ejecuta activator que abre un formulario en el navegador
  4. El usuario ingresa el código de licencia
  5. activator obtiene el fingerprint del hardware (SHA-256)
  6. Envía {codigo, hw_id, email} al Worker (POST /activar)
  7. El Worker valida en Supabase: código existe y no está usado
  8. Si ok: marca como usado, genera archivo .lic firmado con HMAC
  9. activator guarda .lic en el sistema y retorna exit code 0
  10. Si error: muestra mensaje, retorna exit code ≠ 0
  11. El instalador verifica el exit code

Flujo de revalidación

  1. El backend inicia y lee el archivo .lic
  2. Envía .lic + current_hw_id al Worker (POST /revalidar)
  3. El Worker valida: HMAC, hw_id actual vs registrado, estado en Supabase
  4. Si ok: el backend funciona normalmente
  5. Si falla: todas las rutas /api/* devuelven 403, se sirve license-error.html

Capas de seguridad

  1. Primera capa (instalador): activator evita la instalación si el código es inválido
  2. Segunda capa (backend): revalidación contra el Worker cada vez que arranca la app
  3. Firma HMAC: el .lic está firmado, no se puede falsificar sin el secreto del Worker
  4. Binding a hardware: el .lic está vinculado al hw_id, no funciona en otro equipo

Endpoints del Worker

Schema SQL (Supabase)

CREATE TABLE IF NOT EXISTS licencias (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  codigo VARCHAR(32) UNIQUE NOT NULL,
  usado BOOLEAN NOT NULL DEFAULT false,
  hw_id TEXT,
  email TEXT,
  fecha_activacion TIMESTAMPTZ,
  revocado BOOLEAN NOT NULL DEFAULT false,
  creado_en TIMESTAMPTZ NOT NULL DEFAULT now()
);

10. CI/CD y releases

Las releases se generan solo cuando el desarrollador decide versionar, no en cada push.

Disparar una release

git tag v1.0.0          # Elegí la versión
git push origin v1.0.0  # → GitHub Actions se activa

Esto ejecuta el workflow release.yml que:

  1. Compila el backend Go + activator para Linux, Windows y macOS
  2. Empaqueta dist/tpdroid-linux.tar.gz (tarball Linux con install.sh)
  3. Genera dist/TPDroid-Setup.exe (instalador Windows via NSIS)
  4. Crea un draft release en GitHub con ambos archivos

La LICENSE_WORKER_URL se toma de api_admin/.env automáticamente.

Publicar la release

  1. Ir a GitHub → Releases → Drafts
  2. Revisar los artifacts, editar descripción si hace falta
  3. Hacer clic en "Publish release"

Lanzar manualmente

Desde GitHub → Actions → Build and Release → Run workflow (útil para pruebas).
